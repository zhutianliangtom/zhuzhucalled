# 登录设备限制功能实现计划

## 一、需求分析

### 需求描述
当同一账号在多台设备登录时，需要：
1. 新设备登录时检测到已有其他设备登录
2. 提示用户"该账号已在其他设备登录，是否顶号登录"
3. 用户确认后，新设备登录成功，旧设备强制下线
4. 旧设备检测到被顶号后，自动退出登录

### 核心场景
- 用户A在设备1登录 → 用户A在设备2登录 → 设备2提示是否顶号 → 用户确认 → 设备1强制下线

## 二、现有系统分析

### 2.1 认证机制
- **认证方式**: JWT (JSON Web Token)
- **存储方式**: 前端使用 `uni.setStorageSync` 存储 token 和用户信息
- **验证方式**: 后端通过 `authenticateToken` 中间件验证 token

### 2.2 关键文件
| 文件路径 | 功能说明 |
|---------|---------|
| `backend/server.js` | 后端主服务，包含登录接口和认证中间件 |
| `utils/api.js` | 前端 API 封装，包含登录请求和心跳检测 |
| `utils/storage.js` | 前端存储工具 |
| `pages/auth/login.vue` | 登录页面 |

### 2.3 现有登录流程
```
前端提交学号密码 → 后端验证 → 返回 token 和用户信息 → 前端存储 → 跳转到首页
```

## 三、实现方案

### 3.1 架构设计

**核心思路**: 通过维护用户的活跃会话列表，实现设备登录限制

#### 3.1.1 后端架构
- **会话管理**: 使用内存 Map 存储活跃会话（用户ID → 会话列表）
- **Token 黑名单**: 使用 Set 存储已失效的 token
- **登录检测**: 登录时检查该用户是否有其他活跃会话

#### 3.1.2 前端架构
- **顶号检测**: 心跳检测时检查是否被顶号
- **强制退出**: 检测到被顶号后清空存储并跳转登录页

### 3.2 数据库表设计（无需新增表）
- 使用内存存储会话信息，无需修改数据库

### 3.3 API 接口设计

| 接口 | 方法 | 说明 |
|-----|------|-----|
| `/auth/login` | POST | 修改：返回是否有其他设备登录 |
| `/api/heartbeat` | GET | 修改：检测是否被顶号 |
| `/auth/logout` | POST | 新增：主动退出登录 |

### 3.4 数据结构

#### 会话信息
```javascript
{
  token: string,           // JWT token
  deviceId: string,        // 设备标识（可选）
  loginTime: number,       // 登录时间戳
  lastActiveTime: number   // 最后活跃时间
}
```

## 四、详细实现步骤

### 4.1 后端实现（`backend/server.js`）

#### 步骤 4.1.1: 添加会话管理数据结构
```javascript
// 活跃会话存储: { userId: [{ token, deviceId, loginTime, lastActiveTime }, ...] }
const activeSessions = new Map()

// Token 黑名单（已失效的 token）
const tokenBlacklist = new Set()

// 最大允许登录设备数

```

#### 步骤 4.1.2: 修改登录接口 `/auth/login`
- 登录成功前检查该用户是否有其他活跃会话
- 返回 `hasOtherDevice` 标识
- 创建新会话记录

#### 步骤 4.1.3: 修改心跳接口 `/api/heartbeat`
- 检查当前 token 是否在黑名单中
- 检查会话是否被强制下线
- 更新最后活跃时间

#### 步骤 4.1.4: 添加退出登录接口 `/auth/logout`
- 从活跃会话中移除当前会话
- 可选：将 token 加入黑名单

#### 步骤 4.1.5: 添加顶号登录接口 `/auth/login/force`
- 用户确认顶号后调用
- 清除该用户所有旧会话
- 将旧 token 加入黑名单
- 返回新 token

### 4.2 前端实现

#### 步骤 4.2.1: 修改登录页面 (`pages/auth/login.vue`)
- 登录成功后检查是否有其他设备登录
- 如果有，显示确认对话框"该账号已在其他设备登录，是否顶号登录？"
- 用户确认后调用强制登录接口

#### 步骤 4.2.2: 修改 API 封装 (`utils/api.js`)
- 添加 `forceLogin` 方法
- 添加 `logout` 方法
- 修改心跳检测逻辑，检测被顶号情况

#### 步骤 4.2.3: 修改心跳检测回调
- 检测到被顶号时，清空存储并跳转登录页
- 显示提示"您的账号已在其他设备登录"

## 六、流程时序图

### 正常登录流程
```
用户 → 登录请求 → 后端检查无其他设备 → 返回 token → 前端存储 → 跳转首页
```

### 顶号登录流程
```
用户 → 登录请求 → 后端检查有其他设备 → 返回 hasOtherDevice: true
    ↓
用户确认 → 强制登录请求 → 后端清除旧会话 → 返回新 token
    ↓
旧设备心跳 → 检测到 token 失效 → 返回 401 → 前端强制退出
```

---

## 七、异常情况处理

| 异常场景 | 处理方式 |
|---------|---------|
| 后端重启 | 所有会话丢失，用户需要重新登录 |
| token 过期 | 正常过期处理，返回 401 |
| 网络异常 | 心跳检测失败，不强制退出 |
| 用户主动退出 | 从会话列表移除，不影响其他设备 |

---

## 八、安全考虑

1. **Token 黑名单**: 使用 Set 存储失效 token，防止旧 token 被重用
2. **会话超时**: 定期清理长时间不活跃的会话
3. **设备标识**: 可选添加设备标识，便于用户识别登录设备
4. **日志记录**: 记录登录、顶号、退出等操作日志

---

## 九、测试要点

1. **正常登录**: 单设备登录正常
2. **顶号登录**: 第二台设备登录时显示提示，确认后第一台设备被强制下线
3. **心跳检测**: 被顶号后心跳返回失败，前端自动退出
4. **主动退出**: 退出后不影响其他设备登录状态
